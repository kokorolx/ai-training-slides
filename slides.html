Framework

The 7 Levels
of Using AI

Hầu hết dev đang dùng AI như một chatbot thông minh — trong khi phần lớn sức mạnh đang bị bỏ lãng phí mỗi ngày.

Tại sao dừng ở đây?

Memory + Skills + Agents + MCP giải quyết được 99% vấn đề thực tế. RAG và cao hơn là overkill — tốn infra, tốn thời gian, không cần thiết.

1
One-shot chat, copy-paste
Hỏi 1 câu → copy kết quả vào code. Không context, không memory. AI như Google thông minh hơn.
2
Multi-turn, prompt cơ bản
Hội thoại nhiều lượt trong 1 session. Biết viết prompt tốt (few-shot, chain-of-thought). Vẫn mất hết khi đóng tab.
3
Project memory
Hôm nay
CLAUDE.md: AI biết project, tech stack, conventions từ đầu. Không cần giải thích lại. Context persist qua nhiều ngày.
4
Tools + skills + agents
Hôm nay
AI gọi tool (terminal, file, web), chạy workflow tự động. Skills tái sử dụng workflow. Agents tự xử lý multi-step task.
5
RAG
Overkill
AI tra cứu knowledge base nội bộ qua vector search. Cần infra: vector DB + embedding pipeline. Hầu hết team không cần đến đây.
6
Agentic RAG
Overkill
Agent tự quyết định retrieve gì, khi nào. Kết hợp reasoning + retrieval động. Phức tạp, không stable cho production.
7
Full autonomous multi-agent
Overkill
Nhiều agent tự phối hợp, tự phân công, tự debug, tự deploy. Frontier research — chưa production-ready cho hầu hết use cases.

Block 2 · Skills · Example 1

/brainstorming

Use before any feature, component, or behavior change
Checklist enforced
  1. Explore project context (files, docs, commits)
  2. Ask clarifying questions — one at a time
  3. Propose 2–3 approaches + trade-offs + recommendation
  4. Present design section by section, get approval
  5. Write spec doc, commit it
  6. ONLY THEN: implement
❌ Không có skill
Dev: "Add notification badge". AI bắt đầu code ngay. Assume "badge = red dot". Ship → cần work across tabs, clear on click, mobile nav khác → Full rewrite 3h
✅ Có /brainstorming
AI explore → tìm useNotifications composable đã có → hỏi "clear on click?" → propose 3 approaches → spec → implement đúng lần đầu. 45 phút tiết kiệm 3h rewrite.
HARD-GATE: Do NOT invoke any implementation skill or write any code until design is approved

Block 2 · Skills · Example 2

/systematic-debugging

Use when encountering any bug, test failure, or unexpected behavior
IRON LAW: NO FIXES WITHOUT ROOT CAUSE INVESTIGATION FIRST
4 Phases — Cycle
Red Flags — AI hay dùng để skip
✕
"Just try changing X"
→ STOP. Về Phase 1. Chưa có root cause.
✕
"One more fix attempt"
→ STOP. Sau 2+ lần fail — question architecture.
✕
"It's probably X, let me fix"
→ STOP. Assumption không có evidence.
🛑
"3+ fixes failed"
→ STOP. Wrong architecture. Dừng lại, discuss với human.

Với /systematic-debugging

❌ Không có skill — debug theo cảm tính

Trace first → fix once → done
Iron Law: No fixes without root cause investigation first

Vấn đề

Context Rot & Vibe Coding

❌ Vibe Coding
Idea → code ngay, hỏi AI "làm cái này đi"
AI fill context window với debug info, sửa đi sửa lại
Output ngày càng tệ — fix 1 bug tạo ra 2 bug mới
Kết quả: rework, misalignment, mất ngày
✓ Spec-Driven
Brainstorm → spec approve → sau đó mới code
AI làm trong fresh context, không bị rot
Spec = guardrail — AI không drift, không overengineer
Kết quả: đúng ngay lần đầu, ít review cycle
Context rot: càng nhiều code trong context → AI output càng tệ. Spec-driven phòng ngừa bằng cách isolate mỗi phase trong fresh context.
B3 · 3/12
Lý thuyết · Bước 1–2

Brainstorm & Spec

Brainstorm — Explore trước
Hỏi AI "clarify questions" — không phải solutions
Propose 2–3 approaches với trade-offs trước khi chọn
Kết quả: design doc được approve. Tất cả align trước khi code 1 dòng nào.
Spec — WHAT, không phải HOW
❌ BAD SPEC
"Dùng Redis cache TTL 5 phút, invalidate khi user update"
✓ GOOD SPEC
"User profile load < 200ms. Data consistent sau update."
Spec mô tả kết quả, không phải implementation. HOW là việc của tasks.
Good spec = constraint tối thiểu để AI không drift.
B3 · 4/12
Lý thuyết · Bước 3–5

Tasks · AC · Verify

Tasks — Bite-sized
Mỗi task ≤ 30 phút, commit độc lập
Không "implement the feature" — chia nhỏ: schema → API → FE → test
Mỗi task verifiable độc lập
AC — Acceptance Criteria
Mỗi AC là câu hỏi Yes/No
❌
"User có thể login bình thường"
✓
"POST /login valid → 200 + JWT"
Feature done = tất cả AC pass
Verify — Evidence
Không tự claim "xong" — chạy test, show output thật
Screenshot, curl response, test pass count
Evidence before assertions. Always.

MCP — Model Context Protocol

Là gì
MCP là chuẩn kết nối AI với external tools (read + write)
Trước MCP
AI chỉ có text. Muốn AI biết database → phải copy SQL kết quả vào chat.
Sau MCP
AI gọi tool trực tiếp trong conversation. Không cần human làm trung gian.
🔒 Security Model
AI chỉ gọi được những tool bạn cấp phép. Permission scoped theo từng tool.
Architecture
Trước: Human → DB → copy → AI
Sau: AI → MCP → DB → AI (direct)
